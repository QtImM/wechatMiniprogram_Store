# 商品与内容真实化：首批商品接口设计

> 对应 GitHub Issue #10。本文仅定义商品分类、商品列表和商品详情的数据库化改造；首页内容、搜索收藏、评论与交易链路不在本次范围。

## 目标与阶段目标

在小程序 `useMock: false` 时，不改动既有 `catalog/*`、`goods/*` 请求路径和页面调用方式，使分类页、商品列表页、商品详情页均展示数据库中的商品。

| 阶段 | 交付目标 | 完成依据 |
| --- | --- | --- |
| 1 | 分类真实化 | `catalog/index`、`catalog/current`、`catalog/{id}`、`goods/category` 查询数据库 |
| 2 | 列表真实化 | `goods/count`、`goods/list` 支持分类、关键词、分页、新品/热销和排序 |
| 3 | 详情真实化 | `goods/detail`、`goods/related` 返回 SPU、SKU、规格、库存、图片与关联商品 |
| 4 | 数据与回归验证 | `init.sql` 可初始化演示数据；接口验证与 Maven 构建通过 |

完成每个阶段后，更新 `docs/superpowers/status.md`、提交中文 commit 并推送本分支。

## 架构与接口兼容

新增独立的商品应用查询 Controller、Service、Mapper 与响应对象，保留小程序现有的 `/app-api/catalog/**` 和 `/app-api/goods/**` 路径及字段结构。真实接口不再依赖 `MockData`；未迁移的 Mock 接口继续保留在 `AppMockController`，已迁移路径必须移除以避免 Spring 路由冲突。

数据流为：小程序既有页面 → 既有接口路径 → 商品应用查询层 → `product_category`、`product_spu`、`product_sku` → 兼容现有页面的响应对象。

## 数据模型与查询规则

- 分类使用现有 `product_category`，按排序字段构建一级和二级分类结构。
- 列表以已上架 SPU 为主，关联最低可售 SKU 得到展示价格、库存和规格摘要。
- 关键词匹配商品名称与简介；分类筛选包含指定分类及其下级分类；分页参数沿用前端的 `page`、`size`。
- 新品、热销筛选分别以既有 SPU 标识字段和销售排序字段实现；若当前表缺字段，补充最小必要字段及种子值。
- 商品详情返回 SPU 基础信息、轮播图、详情富文本、SKU 列表、规格值与总库存；关联商品优先取同类上架商品，排除当前商品。
- 数据库为唯一商品数据来源，`sql/init.sql` 必须可在空库中建立可浏览的分类、SPU 与 SKU 种子数据。

## 错误处理与兼容约束

- 无效分类、商品不存在或已下架时，返回统一业务错误，不用任意 Mock 商品兜底。
- 分页超出范围返回空列表与正确总数。
- 金额继续使用分为单位的整数；库存由后端返回，不信任前端输入。
- 本次不修改小程序接口常量或页面；若字段缺失，优先在后端响应适配，而非更改前端。

## 测试与验收

1. 使用全新数据库执行 `sql/init.sql`，确认分类、SPU、SKU 种子数据可查询。
2. 逐一验证分类、列表、详情、关联商品接口，并覆盖分类筛选、关键词、新品、热销、分页、无结果和不存在商品。
3. 小程序关闭 Mock 后，验证分类、列表、详情页可正常渲染。
4. 执行 `cd shop-backend && mvn clean install -DskipTests`，结果必须为 `BUILD SUCCESS`。

## 明确不做

首页 Banner、频道、品牌、专题；搜索历史、收藏、浏览足迹、评论；购物车、订单、支付；后台商品管理和鉴权收口，均留给后续 Issue。
